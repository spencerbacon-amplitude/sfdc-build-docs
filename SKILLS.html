Skill vs command vs script. "Skills" live in ~/.claude/skills/<name>/SKILL.md and have rich frontmatter describing when to fire. "Commands" live in ~/.claude/commands/<name>.md and are simpler slash-invocations. Both are surfaced as /<name>. The dashboard treats them identically — pick the right one based on your task, not on where it lives.

Build orchestration build

Four entry paths for code changes. Pick by size + ceremony needs.

SkillTriggersWhen to useOutput
/sfdc-quick-build Jira URL/key alongside "implement this", "build this", "small change", "quick build", "let's just add". Jira is OPTIONAL — conversational asks also fire. 1–8 tasks, all mechanical/standard tier. Want it kicked off + run mostly hands-off. Spec at .claude/specs/<slug>.md + lighter .pm/<slug>/{TODO,QUESTIONS,LOG}.md. Single inline review at end. Q-FINAL gate before merge.
/sfdc-guided-build "help me build", "I want to add", "let's implement", "can we create", "guided build". Conversational features without a Jira ticket. Feature work where design isn't decided yet. LWC mockups, ambiguous scope. Spec at .claude/specs/<slug>.md after conversational design phase. Build proceeds inline.
/sfdc-drive-build Invoked explicitly: /sfdc-drive-build <slug> or /loop /sfdc-drive-build <slug> for unattended runs. ≥8 tasks OR any critical-tier task OR genuine parallel-wave wins. "Leave it running for an hour" style. Spec + full .pm/<slug>/ (adds PROJECT_PLAN, CONTEXT, FINDINGS/). Worker subagents dispatched in parallel waves. Two-stage review per task. Q-PLAN + Q-FINAL gates.
/grill "grill me", "interrogate", "stress-test this plan", asks to be interviewed about an approach. Before any build, when the plan needs sharper edges. Walks the decision tree branch-by-branch and writes the spec when done. Spec at .claude/specs/<slug>.md with frontmatter resolved. Hands off to drive-build (or whichever path is right).

Design design

SkillTriggersWhen to useOutput
/lwc-design "design the [X] component", "let's design an LWC", "draft a design for [component]", "I want to design [feature] before building". Before writing code for any non-trivial LWC. Walks the PICKLES framework (Prototype, Integrate, Composition, Kinetics, Libraries, Execution, Security). Durable design doc at .claude/lwc_design/<componentName>.md. Auto-discovered by quick-build / guided-build when they touch the matching LWC.

Documentation doc

SkillTriggersWhen to useOutput
/design-doc-writer "design doc", "technical design", "write a spec", "document this ticket", "document this change". Auto-fires from quick-build / guided-build / drive-build at completion when spec has jira:. After implementation (or after plan approval). Creates Google Doc in $AMP_DESIGN_DOCS_DRIVE_FOLDER_ID folder. Google Doc URL posted as a comment on the Jira ticket. Updates the existing doc if one is already linked.

Confluence page describing the process as-implemented.

Debugging / investigation util

SkillTriggersWhen to useOutput
/sfdc-debug "X is failing", "why isn't Y working", "debug this", "this is broken", "deploy errors", "test is red". Something's broken and the cause isn't obvious from a single read. Walks deploy errors, test failures, LWC render issues, flow faults, orchestrator BLOCKER loops. Diagnosis + concrete next step. Doesn't auto-fix — surfaces what to change.
/workato-sim /workato-sim <ClassName>, "size this queueable", "will this fit in heap", "what chunk size is safe for <queueable>". Before deploying or expanding the chunk size of a Workato outbound queueable. Sizes payload against 6 MB Apex heap. Measured heap projection at the configured chunk size. Dev-only (refuses sandbox/prod targets).
/spec-view "/spec-view", "open the spec for X", "render the X spec", "show me the spec for X". Read a spec visually instead of the raw markdown. Re-render after editing the .md. .claude/specs/<slug>.html rendered + opened in browser.

Worktree / repo util

SkillTriggersWhen to useOutput
/worktree-cleanup "clean up worktrees", "remove old worktrees", "are there any worktrees I can delete", "audit worktrees". After many merges. Verifies each worktree is clean + merged into master, then removes the safe ones plus their merged branches. Also catches orphan empty dirs. Cleaned worktree list + report of what was removed.
/worktrees "/worktrees" (slash command). Quick snapshot of all current worktrees — branch, dirty count, last commit subject. Read-only inventory. One-screen worktree table.

Decision picker — what should I use?

Most common decisions:
  • I have a Jira ticket and want to ship it → /sfdc-quick-build <TICKET-KEY> (or paste the ticket URL; quick-build will fetch it). If the build has critical-tier tasks or >8 tasks, quick-build will halt and route you to drive-build.
  • I want to build something and the design isn't settled → /sfdc-guided-build (or /grill first if you want to be interrogated through the design decisions).
  • I have a multi-task plan and want to step away → /loop /sfdc-drive-build <slug>.
  • Something is broken → /sfdc-debug.
  • About to build an LWC → /lwc-design <componentName> FIRST.
  • About to deploy a Workato queueable → /workato-sim <ClassName> to validate heap.
  • I edited a spec and want to read it visually → /spec-view <slug>.
  • Trivial single-line change → just describe it conversationally. No skill needed.
