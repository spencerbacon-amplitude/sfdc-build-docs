At-a-glance file comparison

Three modes share the same playbook + always-on rules but differ in how much state they persist. Quick-build and drive-build write to the same .claude/specs/ + .pm/<slug>/ shapes so the dashboard picks them up identically. Conversational writes nothing structural — it's the trivial-change escape hatch.

Conversational mode is intentionally dashboard-invisible. Trivial single-line changes don't justify a spec or state files — they ship in one turn, verified by deploy Succeeded. The only artifact a conversational change leaves behind is a git commit and a row in .pm/DEPLOYS.md from the PostToolUse hook.
Quick-build and drive-build are both dashboard-visible because the dashboard keys off .pm/<slug>/TODO.md existing and the spec frontmatter status: field — both of which they write identically.

Lifecycle timelines (side-by-side)

All three modes converge on a deployed change but differ sharply in ceremony. Conversational is one turn; quick-build is one session; drive-build is an orchestrated loop.

conversational — one turn, trivial change

  1. User describes a trivial change (typo, comment, single-line rename, isolated field-add with no consumers)
  2. Create worktree (CLAUDE.md hard rule — required for any code edit, even one-line)
  3. Edit · deploy
  4. Verification = deploy Succeeded
  5. Commit · merge to master · cleanup worktree

No spec, no .pm/<slug>/, no Q-FINAL, no dashboard card. The only durable artifact is the git commit + .pm/DEPLOYS.md row from the PostToolUse hook.

quick-build — sequential, one session

  1. Read Jira (if any) OR conversational ask
  2. Explore subagent → file inventory
  3. Plan presentation + tier eligibility check (any critical → halt, route to drive-build)
  4. User approves plan conversationally (this IS the plan gate)
  5. Write .claude/specs/<slug>.md · create .pm/<slug>/{TODO,QUESTIONS,LOG}.md · render spec .html
  6. Create worktree · flip spec status: in-progress
  7. Implement tasks · mirror TaskList → TODO.md on each transition · append to LOG.md
  8. Deploy · verification gate · single inline review pass
  9. Design doc (if Jira)
  10. Write Q-FINAL to QUESTIONS.md
  11. User answers Q-FINAL with merge it
  12. Re-render spec .html · stage + commit · merge to master · close Q-FINAL · post Jira comment (if Jira) · flip spec status: deployed

drive-build — parallel waves, multi-iteration

  1. Resolve slug · check spec exists (chain to /grill if not)
  2. Read .claude/specs/<slug>.md · pre-research dispatch → CONTEXT.md
  3. Plan: write PROJECT_PLAN.md + TODO.md with shape: classifications
  4. Risk-review Explore → append to PROJECT_PLAN.md ## Risks to discuss
  5. Write Q-PLAN to QUESTIONS.md
  6. User answers Q-PLAN with approval
  7. Wave dispatch: fan out workers in parallel · each gets sliced playbook excerpts + Playbook focus: pointer per shape
  8. Workers return DONE → verification gate per task → [r]
  9. Two-stage review parallel: spec (Sonnet) + quality (Haiku); CHANGES_REQUESTED → findings to FINDINGS/<T-NNN>-r<N>.md → re-dispatch fix worker
  10. Drift audit: check picklist consumer reachability + spec drift → Q-DRIFT if anything fires
  11. Write Q-FINAL to QUESTIONS.md
  12. User answers Q-FINAL with approval
  13. Merge all worker branches in dependency order · close Q-FINAL · jira-deploy-comment + design-doc-writer · flip spec status: deployed

Status state machines

Spec frontmatter status: quick-build drive-build

ready-to-build→ in-progress→ deployed→ done

ready-to-build: spec written, build not started. in-progress: worktree exists, code work happening. deployed: merged to master, awaiting Copado prod promotion. done: prod-live, fully complete (human flips this post-Copado). Also: planning for scoping-only specs; shipped is a backwards-compat alias for deployed. Conversational mode has no spec — no status to track.

Task status in TODO.md

[ ] ready→ [~] running→ [r] review→ [x] done

Quick-build: skips [r] — single inline review at end. Drive-build: uses all four; [r] flips back to [~] on CHANGES_REQUESTED. Also [?] for BLOCKER/blocked.

Question lifecycle in QUESTIONS.md

written under 🔴 OPEN→ awaiting answer→ user types under ### ANSWER→ moved to ✅ RESOLVED

Stop hook T2 counts "awaiting" strings per build, compares to .last_pinged_count. New count > last → Slack ping. Skill resolves by moving the block to RESOLVED + appending date + answer text.

Body sections (canonical)

.pm/<slug>/TODO.md both

Task list · rewritten on every TaskCreate / TaskUpdate

Format

# TODO — <slug>

Statuses: [ ] ready · [~] in progress · [r] in review · [?] blocked · [x] done

- [x] T-001 (sfdc-metadata-worker) — Add Custom_Foo__c field on Account
  files: force-app/main/default/objects/Account/fields/Custom_Foo__c.field-meta.xml
  success: deploy returns Succeeded
- [~] T-002 (sfdc-apex-worker) — depends:T-001 — Read Custom_Foo__c in handler
  files: force-app/main/default/classes/AccountTriggerHandler.cls
  shape: apex-trigger-handler-edit, apex-soql-edit
  success: deploy + test pass, coverage ≥80%

Quick-build vs drive-build

Same format. Quick-build statuses cycle through [ ] → [~] → [x] (single review pass, no [r] phase). Drive-build adds [r] for tasks under spec/quality review, and may use [?] for worker BLOCKER results.

Consumed by

Drive-build dashboard (parses task counts + statuses for the Active card), SessionStart hook S2 (prints first 40 lines on resume).

.pm/<slug>/PROJECT_PLAN.md drive-build only

Plan + risk review · written once at Step 2a, updated only if plan revises

Why drive-build needs this and quick-build doesn't

Drive-build runs a risk-review Explore subagent on the plan, persists the findings under ## Risks to discuss, and gates on user approval via Q-PLAN. Quick-build skips both — for ≤8 mechanical/standard tasks the conversational plan presentation at Step 3 IS the plan + risk review + gate.

Sections

Goal · Approach · Scope boundaries · Risks to discuss (filled by Explore) · Open items resolved.

.pm/<slug>/CONTEXT.md drive-build only

Research dump · written once or appended on gap-fills

Why drive-build needs this

Workers dispatched in parallel waves can't share session memory. CONTEXT.md is the shared codebase grounding that each worker reads at dispatch — it holds full file contents from sfdc-research-worker so workers don't have to re-explore.

Why quick-build doesn't

Single session, no worker fanout. Research stays in the conversation context where it was gathered.

.pm/<slug>/FINDINGS/<T-NNN>-r<N>.md drive-build only

Per-task review findings · one file per (task × review-round)

Why drive-build needs this

Two-stage review (spec + quality) runs per task in parallel. When a review returns CHANGES_REQUESTED, the findings need to be persisted so the fix worker can read them at dispatch — they can't be passed via session memory because the fix worker is a fresh Agent.

Quick-build's equivalent

The single inline review pass happens in-session before Q-FINAL. Any findings are addressed conversationally and refixed in the same session — no persistence needed.

.pm/DEPLOYS.md both — auto

Repo-root deploy ledger · PostToolUse hook P1 writes it

Every sf project deploy start appends <ts> | <status> | <org> | <deployId>. Skill-agnostic — fires on any deploy, regardless of which skill triggered it.

Why this split exists

Three modes, three ceremony levels. The split exists because not every change deserves the same amount of structure:
  • Conversational — one-turn trivial changes. No spec, no .pm/. Ceremony cost would exceed the change.
  • Quick-build — small builds (1–8 tasks, all mechanical/standard tier). Drops the Q-PLAN gate, the worker subagent fanout, and the per-task two-stage review with persisted findings — one inline review at end is enough.
  • Drive-build — multi-task or critical-tier work. Worth the orchestrator, parallel workers, two-stage review per task, and persisted findings because the blast radius justifies it.
The tier check at quick-build Step 3 is the safety valve. Any task that maps to a critical-tier shape in drive-build's playbook focus table (TriggerDispatcher edits, picklist-value-add with consumer fanout, shared-LWC @api changes, picklist-consumer LWC edits) forces a halt and a route to /sfdc-drive-build. The trade is "fast for safe work" — not "fast at all costs."

What all three share

  • Worktree-mandatory for any code edit (CLAUDE.md hard rule, enforced by PreToolUse hook R5 on force-app/)
  • Playbook compliance per docs/sfdc-playbook.md
  • Verification gate — at minimum deploy Succeeded
  • PostToolUse deploy logger appends to .pm/DEPLOYS.md

What quick-build and drive-build share (beyond the above)

  • Spec at .claude/specs/<slug>.md with consistent frontmatter shape
  • .pm/<slug>/ state dir so dashboard renders both identically
  • Q-FINAL human-gate before merge
  • Verification gate before review (deploy + tests + diff scope)
  • Two-stage logic — spec compliance first, then code-quality playbook check (drive-build runs them as parallel sub-agent dispatches; quick-build runs them inline back-to-back)
  • Same merge ritual: re-render spec .html, pre-merge worktree audit, --no-ff merge, worktree cleanup

The "kick to drive-build" path

If quick-build's tier check at Step 3 finds a critical task or >8 tasks, it writes the spec with build_skill: sfdc-drive-build and halts with a visible re-fire reminder. The user then runs /sfdc-drive-build <slug> (or /loop /sfdc-drive-build <slug>) and drive-build picks up from the spec — no duplicated work.
