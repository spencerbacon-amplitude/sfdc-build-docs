Why shapes exist

Workers dispatched in parallel waves can't share session context. Each one gets a fresh prompt with sliced playbook excerpts. To make those prompts useful, the orchestrator needs to know what kind of task each is so it can paste the right Playbook focus pointer — a one-line reminder about the recurring review findings for tasks of that shape.

Each task in TODO.md can carry a shape: line listing one or more shape keys. The drive-build dispatcher looks each up in the table below and pastes the matching Pointer verbatim into the worker's brief. No per-iteration re-classification.

Quick-build uses the same table for eligibility. If any planned task has a critical-tier shape, quick-build halts and routes to /sfdc-drive-build instead. The shape system is the shared safety boundary between the two skills.

Tier definitions

Each shape carries a tier. New shapes get classified by applying these criteria in order — first match wins:

mechanical

Single-pattern XML or one-shape edit. No judgment calls. No consumers to enumerate. Failure mode caught by deploy Succeeded.

Example: adding a custom field XML, extending a layout, adding FLS to an operator-facing perm set.

standard

Needs taste/judgment (test assertion choice, SOQL placement, button styling) but blast radius is the file you touched. Failure mode caught by deploy + tests + diff-scope.

Example: extending an existing trigger handler, adding a new selector method, writing an LWC's Cancel/Confirm button.

critical

Touches shared framework OR has consumer fanout. Failure mode requires planner-time enumeration of every consumer / handler / caller. Deploy + tests can pass while prod still breaks.

Example: editing TriggerDispatcher, adding a new picklist value, changing an @api surface on a shared LWC.

Any critical task in a build disqualifies the build from sfdc-quick-build. The build must route to /sfdc-drive-build instead. This is the hard safety valve.

Classifying a borderline task

If a task could read as two tiers, apply the three criteria in order:

  1. Is it a single-pattern XML or one-shape edit with no judgment calls? → mechanical.
  2. Does it need taste/judgment (assertion choice, SOQL placement, styling) but stay within the file you touched? → standard.
  3. Does it touch shared framework OR have consumer fanout? → critical.

When in doubt, classify higher. A standard task running through critical-path review wastes a few minutes; a critical task running through standard review can silently ship broken consumers.

Adding a new shape

If you find yourself classifying a task and no existing shape fits — that's a signal to add one. Process:

  1. Identify the recurring review finding. Shapes earn their place by being review-cycle problems that come up more than once. If you've reviewed three tasks of the same kind and caught the same issue, that's a shape.
  2. Pick a tier using the three criteria above.
  3. Write the pointer. One sentence, imperative voice, names the specific failure mode and the corrective rule. Example pattern: X discipline → do Y, never Z (recently caught: ...).
  4. Edit ~/.claude/skills/sfdc-drive-build/SKILL.md — add the row to the playbook focus pointers table (around line 320). Also add the shape key to the bullet list above the table.
  5. Update SHAPES.html (this doc) — that's how it shows up in the dashboard.
  6. If critical: update the CLAUDE.md "Critical-tier shapes that force drive-build" line so the user-facing decision tree stays accurate.
