Authoritative coding rules for this Salesforce DX project. Every entry path (conversational, sfdc-quick-build, sfdc-guided-build, /sfdc-drive-build workers) follows these rules.

If a rule conflicts with something I'm about to write, I stop and surface the conflict — I don't "just write it and hope". Most of these are deploy- or runtime-blockers.

Apex hard-stops

Refuse to write code that violates any of these. If the task seems to require violation, stop and ask.

  • SOQL / DML inside loops. Includes indirect cases (a method called in a loop that itself does SOQL/DML). Bulkify, batch, or hoist out of the loop.
  • @future is forbidden. Use Queueable (supports chaining and non-primitives) or Batch (large volumes). No exceptions without an explicit written task instruction.
  • WITH USER_MODE on SOQL that touches user-visible data. For DML where partial failure is recoverable, use Database.update(records, false) and inspect Database.SaveResult[]. Same pattern for insert / upsert / delete.
  • No System.debug() in hot paths. It consumes CPU even when debug logging is inactive. Acceptable in test methods and one-off scripts; never in trigger handlers, queueables, batches, or @AuraEnabled methods.
  • No hardcoded IDs. Use Custom Metadata, Custom Settings, Custom Labels, or Schema.getGlobalDescribe() lookups. Record IDs vary between sandboxes and prod.
  • Bind variables in dynamic SOQL — always. Database.query('SELECT ... WHERE Foo = :bar') with bar as a typed Apex variable. Never concatenate user input into a query string. For dynamic field names (not values), validate against an allowlist or Schema.SObjectField map.
  • PII fields must be encrypted at rest. If a task adds a new PII field, confirm encryption strategy before writing reads/writes.
  • Missing explicit with sharing / without sharing on new classes. inherited sharing is acceptable when chosen deliberately; the keyword must be present.
  • SOQL without LIMIT or with broad/no WHERE. If the result set could grow unbounded, add a LIMIT or paginate.

Apex authoring patterns

For new classes (not modifying existing ones), prefer this layering. If existing helpers already split responsibilities differently in the codebase, match the existing pattern — don't introduce a new layering style mid-project.

Type Default sharing Role
{SObject}Service with sharing Orchestration; delegates SOQL to selectors, validation/derivation to domains
{SObject}Selector inherited sharing SOQL only; returns List<SObject> or Map<Id, SObject>
{SObject}Domain with sharing Validation, defaults, derivations; no SOQL/DML
{Descriptive}Queueable with sharing Enqueueable job; chainable; supports non-primitives
{Descriptive}Batch with sharing Database.Batchable<SObject> for large datasets
{Descriptive}Util with sharing Stateless helpers
{Descriptive}Exception (n/a) Custom exception

Async decision: Queueable is the default. Batch when you must process more rows than a Queueable's heap allows. Scheduled Flow preferred over Schedulable Apex for recurring work. Finalizer for post-job cleanup. Never @future.

Methods + variables: camelCase. Constants: UPPER_SNAKE_CASE, declared private static final.

They're starter shapes only — the README lists the mandatory project deltas (System.debug removal, WITH USER_MODE on selectors, ApiCalloutLogger for queueable observability, etc.) you must apply before deploying.

SOQL discipline

Curated Salesforce reference: docs/data-references/querying-soql/ — semi-joins, polymorphic relationships, bulk patterns.

Most Apex performance and security incidents come from sloppy SOQL.

Selectivity

A query is selective if it returns <10% of the first million matching records (or <5% beyond), or filters on an indexed field. Non-selective queries on large objects (Account, Contact, Opportunity, Task, Event) will time out or trigger QueryException: Non-selective query against large object type.

Indexed by default:

  • Id, Name, OwnerId, CreatedDate, LastModifiedDate, SystemModstamp, RecordTypeId
  • All External ID fields and Master-Detail relationship fields
  • Lookup fields with unique constraints
  • Standard-object specifics: Account.AccountNumber, Account.Site, Contact.Email, Lead.Email, Case.CaseNumber

If a WHERE clause has no indexed field and the target object has >100K rows, request a custom index from Salesforce Support — or rethink the design.

Wildcards: LIKE 'Acme%' uses the index. LIKE '%corp' and LIKE '%acme%' do not (table scan). Trailing wildcards only.

Inspect the query plan when unsure

sf data query --query "SELECT Id FROM Account WHERE Industry = 'Tech' AND CreatedDate = LAST_N_DAYS:30" --use-tooling-api --plan --target-org spencerdev

LeadingOperationType: Index is good. TableScan means rewrite the filter.

Security mode — required on every query

Pick one explicitly per query — the platform will not infer:

  • WITH USER_MODE (preferred default for user-initiated paths) — enforces both FLS and record-level sharing. Apex 2024+ standard.
  • WITH SECURITY_ENFORCED — enforces FLS, throws System.QueryException on inaccessible fields. Use when you'd rather fail loudly than hide rows.
  • WITH SYSTEM_MODE — bypasses FLS and sharing. Required (not optional) for trigger handlers and system bookkeeping — see next subsection.

For DML where partial failure should not abort the transaction, pair with Security.stripInaccessible(AccessType.READABLE | UPDATABLE, records) to silently drop fields the user can't write.

When SYSTEM_MODE is mandatory — trigger handlers + system bookkeeping

Hard rule: SOQL and DML in trigger handlers, denormalization queueables, and scheduled bookkeeping jobs MUST use WITH SYSTEM_MODE (and AccessLevel.SYSTEM_MODE for Database.update / insert / upsert / delete). This is enforced by the pre-commit grep at scripts/check-trigger-handler-system-mode.js for *TriggerHandler.cls and by sfdc-review-worker Pass 2 for everything else.

Why: trigger handlers fire for every DML, regardless of who initiated it. If a user without FLS on a derived field (Copado deploy user, integration user, anyone without the relevant perm set) inserts or updates a parent record, a WITH USER_MODE query inside the handler throws System.QueryException and rolls back the entire transaction. This blocks deploys and breaks integrations. The propagation is the system's responsibility, not the user's — SYSTEM_MODE is correct.

Applies to:

  • *Trigger.cls thin triggers and *TriggerHandler.cls / *TriggerHelper.cls handler classes.
  • Queueables / batches / schedulables that exist to propagate denormalized fields, run backfills, or keep derived state in sync (e.g. AlertNameBackfillQueueable).
  • Database.update(records, AccessLevel.SYSTEM_MODE) — not AccessLevel.USER_MODE — for the DML that follows a SYSTEM_MODE read in the same code path.

Does NOT apply to:

  • @AuraEnabled controller methods called from LWC — user-facing, USER_MODE is correct.
  • REST resources (@RestResource) — user-facing.
  • Invocables called from screen flows where the user's permissions should be respected.
  • Selectors used by both user-facing and system code — pass AccessLevel as a parameter so each caller chooses.

Sanity check before writing WITH USER_MODE in any new class: ask "would a deploy with RunLocalTests succeed for a Copado service user who has no perm sets beyond the integration default?" If the answer is no, the code is in a system path and needs SYSTEM_MODE.

Bulk patterns — non-negotiable

  • Never SOQL inside a loop. Pull needed records into a Map<Id, SObject> before the loop, then map.get(id) inside.
  • SOQL for loops for large datasets — auto-batches in 200-record chunks, avoids heap blowout: apex for (Account a : [SELECT Id, Name FROM Account WHERE Industry__c = 'Tech' WITH USER_MODE]) { // process one at a time, heap stays bounded }
  • Bulk lookup pattern for triggers: apex Set<Id> accountIds = new Set<Id>(); for (Opportunity o : Trigger.new) accountIds.add(o.AccountId); Map<Id, Account> accountById = new Map<Id, Account>([ SELECT Id, Owner.Email FROM Account WHERE Id IN :accountIds WITH USER_MODE ]); for (Opportunity o : Trigger.new) { Account a = accountById.get(o.AccountId); // O(1) }

Dynamic SOQL

When the query must be constructed at runtime, use Database.queryWithBinds (modern, injection-safe): 

Map<String, Object> binds = new Map<String, Object>{
    'industry' => industry,
    'minRev'   => minRev
};
List<Account> rows = Database.queryWithBinds(
    'SELECT Id, Name FROM Account WHERE Industry = :industry AND AnnualRevenue >= :minRev WITH USER_MODE',
    binds,
    AccessLevel.USER_MODE
);

For dynamic field names (not values), validate against Schema.SObjectType.<Obj>.fields.getMap() keys.

SOQL anti-patterns — refuse to write

  • Equivalent of SELECT * — querying every field "just in case". Breaks FLS surface area.
  • Querying with no WHERE on a custom or standard object with >10K rows.
  • Filtering in Apex after a broad query. Push the filter into the WHERE.
  • Calling a method that does SOQL/DML inside a loop. Most-missed governor violation.
  • Hardcoded ID literals in WHERE clauses.

Selector pattern

If adding more than 2 queries against the same object, lift them into a selector:

public inherited sharing class AccountSelector {
    public static Map<Id, Account> byIds(Set<Id> ids) {
        if (ids == null || ids.isEmpty())
            return new Map<Id, Account>();
        return new Map<Id, Account>(
            [
                SELECT Id, Name, Industry, OwnerId
                FROM Account
                WHERE Id IN :ids
                WITH USER_MODE
            ]
        );
    }
}

Centralizes field lists, makes FLS audit a one-class job, and lets tests mock via @TestVisible static List<Account> mockData. Match existing selector patterns in the codebase before introducing a new style.

Apex test discipline

Curated Salesforce reference: docs/debug-references/running-apex-tests/ — test invocation, coverage flags, red-test debugging.

A passing test run that violates these is still a fail — rewrite before claiming DONE.

Starter scaffolds + reference patterns: docs/apex-test-templates/ has a *Test.cls scaffold (positive / negative / bulk slots, modern Assert API) plus pattern catalogs for assertions, async testing, mocking, and data factories. Same rule as the Apex templates — the README lists mandatory project deltas (AMPTestDataFactory, TriggerSettings__c disables, named mocks) that must be applied before the file is deployable.

  • One behavior per test method. Positive, negative, and bulk are separate methods. testFoo_NullInput and testFoo_EmptyInput are two methods, not one.
  • Bulk tests use 251+ records, not 200. A 200-record test passes inside a single trigger batch; you need to cross the boundary to prove bulkification. For Database.Batchable tests, set batchSize >= testRecordCount — only one execute() invocation runs in test context.
  • Use the Assert class, not System.assert*. Assert.areEqual(expected, actual, msg), Assert.isTrue(cond, msg), Assert.isNotNull(x, msg), Assert.fail(msg).
  • Assert exact values whenever the value is deterministic. Never assert(count > 0) if you know the count is 3 — assert 3.
  • Test the negative paths. Every method with branching needs a test that hits the failure branch. Exceptions: catch and assert the type, don't let them propagate.
  • Wrap async + governor-sensitive work in Test.startTest() / Test.stopTest(). Resets governor limits; forces async to execute synchronously inside the block.
  • Never seeAllData=true. Build data via AMPTestDataFactory.
  • Mock external boundaries. Test.setMock(HttpCalloutMock.class, ...) (existing mocks: WorkatoCalloutMock, GainsightCompanyGSIDHttpMockResponse, MavenlinkProjectMockService). Test.setFixedSearchResults(...) for SOSL.
  • @TestSetup over per-method setup when the same fixture serves multiple tests.
  • Coverage gate: ≥80% on the changed class. Read coverage from the test result, not the org-wide aggregate.

If a test only asserts "no exception thrown" — that's not a test, that's a smoke check. Add an assertion against actual data the method produced.

Testing USER_MODE selectors / services

When the class under test runs SOQL or DML with AccessLevel.USER_MODE (or WITH USER_MODE / WITH SECURITY_ENFORCED / insert as user), the test must run inside System.runAs(testUser) where testUser has the gating permission set assigned before the test session starts. A mid-@TestSetup PermissionSetAssignment for the running user does NOT refresh the session FLS cache — the @IsTest SOQL still throws "No such column" for fields the assignment was supposed to grant. The only reliable pattern is a dedicated user provisioned in @TestSetup, run as via System.runAs(...) in each @IsTest.

Canonical example: PodAlertSelectorTest. Pattern: 

@IsTest
private class FooSelectorTest {
    private static final String TEST_USER_USERNAME_PREFIX = 'fooFls_';

    @TestSetup
    static void makeData() {
        // 1. Mint the test user + assign the perm set. Wrap in runAs(currentUser) to
        //    avoid MIXED_DML when the @TestSetup later inserts non-setup SObjects.
        User testUser;
        System.runAs(new User(Id = UserInfo.getUserId())) {
            testUser = new User(
                Username = TEST_USER_USERNAME_PREFIX + System.currentTimeMillis() + '@amplitude.com.invalid',
                Email = 'foo.fls@amplitude.com', // real domain — satisfies email-domain validation rule
                Alias = 'fooFls',
                FirstName = 'Foo',
                LastName = 'FlsTest',
                TimeZoneSidKey = 'America/Los_Angeles',
                LocaleSidKey = 'en_US',
                EmailEncodingKey = 'UTF-8',
                LanguageLocaleKey = 'en_US',
                ProfileId = [SELECT Id FROM Profile WHERE Name = 'Standard User' LIMIT 1]
                .Id
            );
            insert testUser;
            List<PermissionSet> permSets = [SELECT Id FROM PermissionSet WHERE Name = 'Foo_Access' LIMIT 1];
            if (!permSets.isEmpty()) {
                insert new PermissionSetAssignment(AssigneeId = testUser.Id, PermissionSetId = permSets[0].Id);
            }
        }

        // 2. TriggerSettings__c is a User-hierarchy custom setting. Insert a row for
        //    BOTH the deploying user AND testUser — triggers fire in the inserting
        //    user's context, and code running inside runAs(testUser) reads testUser's row.
        insert new List<TriggerSettings__c>{
            new TriggerSettings__c(SetupOwnerId = UserInfo.getUserId(), AccountTriggerDisabled__c = true /*, ...*/),
            new TriggerSettings__c(SetupOwnerId = testUser.Id, AccountTriggerDisabled__c = true /*, ...*/)
        };

        // 3. Insert your test data. If the class under test does `insert as user` of a
        //    child with a foreign key, reassign the parent record's OwnerId = testUser.Id.
        //    viewAllRecords on the perm set covers SOQL reads but NOT DML cross-reference;
        //    ownership is the reliable path.
        Account acc = AMPTestDataFactory.createAccount();
        acc.OwnerId = testUser.Id;
        update acc;
    }

    private static User getTestUser() {
        return [SELECT Id FROM User WHERE Username LIKE :(TEST_USER_USERNAME_PREFIX + '%') LIMIT 1];
    }

    @IsTest
    static void testFoo() {
        User u = getTestUser();
        System.runAs(u) {
            Test.startTest(); // keep start/stop INSIDE runAs so governor resets apply under testUser
            // ... existing test body ...
            Test.stopTest();
        }
    }
}

Gotchas to remember (each one bit us once during the GTMSYS-12468 rollout):

  • viewAllRecords on the perm set is not enough for DML cross-reference. If MyService.cls does insert as user vo with vo.Account__c = something, testUser needs ownership of something (or a sharing row), not just viewAllRecords. Reassign the parent's OwnerId = testUser.Id in @TestSetup for any record that becomes a foreign key target in a USER_MODE insert/update.
  • Per-test inserts in the @IsTest method need the same ownership treatment. If the test body inserts an Opportunity and later calls a service that does insert as user vo with vo.Opportunity__c = opp.Id, set opp.OwnerId = testUser.Id at insert time.
  • Username convention: Email = '...@amplitude.com' (satisfies the org's email-domain validation rule), Username = '...@amplitude.com.invalid' + System.currentTimeMillis() (globally unique across all orgs; the .invalid TLD avoids collisions with real prod usernames).
  • Per-class username prefix. Use a unique short prefix per test class (paFls_, araFls_, msFls_) so the getTestUser() LIKE query never collides with another class's user.
  • When tests filter by current-user ownership (e.g., byUserId(UserInfo.getUserId())), reassign the seeded child records to testUser at the end of @TestSetup so the filter still resolves inside runAs(testUser).

When NOT to apply this pattern: if the class under test only uses USER_MODE on fields that every base profile already reads (e.g. Name, Status__c, base Account fields), the original "current user" pattern is fine. The pattern is for selectors/services that read fields gated behind a perm set.

Apex callouts — self-callouts to the org's own REST / Tooling API

When an @AuraEnabled method needs to read metadata that's only available via the Tooling API (e.g. PathAssistant, FieldDefinition filters that regular SOQL rejects), there are several platform gotchas that will burn hours if you discover them at integration time. The pattern below is the canonical workaround; reference implementation lives in PathAssistantConfigController, PathAssistantSessionProvider.page, and lwc/picklistPath/.

UserInfo.getSessionId() returns the wrong token from @AuraEnabled context

Calling UserInfo.getSessionId() works as expected from anonymous Apex (returns the 112-char OAuth token usable against /services/data/). The same call from inside an @AuraEnabled method invoked by an LWC returns a Lightning frontdoor token of the same length that the REST API rejects with 401 INVALID_SESSION_ID ("This session is not valid for use with the REST API").

Auth.SessionManagement.getCurrentSession().get('SessionId') is NOT a workaround — it returns the 18-char AuthSession record ID, not a Bearer token. Same 401.

Workaround (documented Salesforce pattern): a tiny Visualforce page that renders {!$Api.Session_ID}. The LWC loads the page in a hidden iframe; the page posts the API-valid session token to its parent via postMessage; the LWC catches the message and passes the token as a parameter to an Apex method that uses it as the Bearer header. Cross-origin fetch('/apex/...') is blocked by CORS in Lightning Experience, so the iframe + postMessage pattern (not fetch) is the path that actually works.

Reference VF page (PathAssistantSessionProvider.page):

<apex:page contentType="text/html" showHeader="false" sidebar="false" standardStylesheets="false" cache="false">
    <script type="text/javascript">
        window.parent.postMessage({ sfPathAssistantSession: '{!$Api.Session_ID}' }, '*');
    </script>
</apex:page>

Reference LWC hookup (in connectedCallback or property setter):

let cachedApiSessionId = null;  // module-scope cache survives across LWC instances on the same tab

_loadAsync() {
    if (cachedApiSessionId) { this._callApex(cachedApiSessionId); return; }
    this._messageHandler = (event) => {
        const sid = event?.data?.sfPathAssistantSession;
        if (!sid) return;
        cachedApiSessionId = sid;
        window.removeEventListener('message', this._messageHandler);
        this._needsSession = false;
        this._callApex(sid);
    };
    window.addEventListener('message', this._messageHandler);
    this._needsSession = true;  // renders the iframe
}

disconnectedCallback() {
    if (this._messageHandler) window.removeEventListener('message', this._messageHandler);
}

Reference Apex:

@AuraEnabled
public static String getConfigJson(Id recordId, String sessionId) {
    String effectiveSid = String.isNotBlank(sessionId) ? sessionId : UserInfo.getSessionId();
    HttpRequest req = new HttpRequest();
    req.setEndpoint(URL.getOrgDomainUrl().toExternalForm() + '/services/data/v66.0/tooling/query/?q=' + EncodingUtil.urlEncode(soql, 'UTF-8'));
    req.setHeader('Authorization', 'Bearer ' + effectiveSid);
    req.setHeader('Content-Type', 'application/json');
    req.setMethod('GET');
    req.setTimeout(5000);
    // ... send and parse
}

Cache the session at module scope in the LWC — subsequent record pages in the same browser tab skip the iframe round-trip entirely.

@AuraEnabled(cacheable=true) is incompatible with HTTP callouts

Adding cacheable=true to an @AuraEnabled method that performs an HTTP callout throws a system error. Drop cacheable=true when you need a callout and cache on the client side instead (module-scope variable in the LWC, or sessionStorage if you need cross-component sharing).

Aura's serialization of Map<String, X> hides keys from the JS side

When an @AuraEnabled method returns a class with @AuraEnabled public Map<String, List<String>> stageToFields, the response object on the JS side has stageToFields: {...} but Object.keys(result.stageToFields) returns [] — the keys are present in the runtime representation but not as enumerable own properties. Direct lookup by string key (result.stageToFields['1 - Discovery']) returns undefined.

Workaround: add a mirror field on the same class that holds the JSON-serialized map:

public class PathConfig {
    @AuraEnabled
    public Map<String, List<String>> stageToFields; // kept for Apex callers
    @AuraEnabled
    public String stageToFieldsJson; // for LWC consumption
    // ... populate stageToFieldsJson = JSON.serialize(stageToFields) before returning
}

LWC does JSON.parse(result.stageToFieldsJson) once and uses the parsed plain object normally.

If returning a top-level String from @AuraEnabled (e.g. getConfigJson returning JSON.serialize(cfg)), the LWC parses the whole payload once — this also sidesteps any field-level schema caching the Aura framework does (a separate gotcha where a newly-deployed @AuraEnabled field can be silently stripped from cached responses until the user re-logs).

Tooling SOQL filter quirks

The relationship names that work in regular SOQL don't all work in Tooling. Two that bit us during the picklistPath build:

  • PathAssistant filtering — use WHERE SobjectType = 'Opportunity'. Do NOT use WHERE EntityDefinition.DeveloperName = 'Opportunity' — Tooling rejects it with "Didn't understand relationship 'EntityDefinition'".
  • Metadata field requires single-row results — selecting Metadata (or FullName) on a Tooling object requires a query that returns exactly one row. Multi-row queries with Metadata in SELECT throw "the query qualifications must specify no more than one row". The workaround is a two-step pattern: query for matching IDs first, then GET /services/data/v66.0/tooling/sobjects/<Type>/<Id> to fetch each one with its full Metadata payload.

Remote Site Settings are NOT required for self-callouts to the org's My Domain

The platform allows callouts to URL.getOrgDomainUrl().toExternalForm() + '/services/data/...' implicitly. No remoteSiteSettings/*.remoteSite-meta.xml deploy needed.

Code analyzer pass

After tests pass, run the Salesforce code analyzer against changed files and remediate findings before deploying: 

sf code-analyzer run --target force-app/main/default/classes/Foo.cls --target force-app/main/default/classes/FooTest.cls
  • sev1 / sev2 (Critical / High): must fix before DONE.
  • sev3 (Medium): fix if it's in the file you touched; note it if pre-existing.
  • sev4 / sev5 (Low / Info): log and proceed unless trivially fixable.

If analyzer isn't installed (sf plugins inspect code-analyzer returns nothing), skip and note "code-analyzer unavailable" — don't fail the task on tooling absence.

LWC hard-stops

Starter scaffolds + PICKLES framework: docs/lwc-templates/ has 13 component bundles (basic, datatable, modal, form, flow-screen, graphql, record-picker, workspace, state-store, LMS, TypeScript, jest, apex-controller) and 16 reference docs. The PICKLES architectural framework — Prototype, Integrate, Composition, Kinetics, Libraries, Execution, Security — lives in references/component-patterns.md and references/lwc-best-practices.md, and /lwc-design walks PICKLES branch-by-branch when designing a new component. The 165-point quality rubric in references/scoring-and-testing.md is the reference checklist for review. As with the Apex starters, the README lists mandatory project deltas (native buttons over lightning-button for action buttons, Amplitude color tokens, WorkatoService for outbound, etc.) — apply before deploy.

If the task requires one of these, stop and surface the conflict.

  • Never mutate @api properties. Copy first: this.items = [...this.items, newItem]. Mutating @api is silent corruption.
  • Never combine @api with @track or @wire. Decorate getters only (never the setter) for @api getter/setter pairs.
  • Never @track a primitive. Primitives have been reactive by default since Spring '20. @track is only for nested-property mutation on objects/arrays — and even then, reassign the whole object when you can.
  • Never use if:true / if:false. Use lwc:if / lwc:elseif / lwc:else.
  • Never use index as the for:each key. Use a stable unique ID (key={item.Id}).
  • Never call @wire with a non-reactive parameter. Use '$propName' with propName as @api or tracked. A literal or local let pins the wire to its initial value.
  • Never forget cleanup in disconnectedCallback. Every addEventListener, every subscription, every interval — pair with removal.
  • Never override .slds-* classes directly. Use SLDS styling hooks (--slds-c-button-brand-color-background: ...).
  • Never select base components by tag in CSS (lightning-button { ... }). Shadow DOM blocks it.
  • Never rely on compiler-generated scope tokens (c-child[lwc-2j48...-host]). Use :host for component-level styles.

LWC modern patterns (API 66.0)

The project is on API 66.0 — these features are available; use them when they reduce code or improve readability.

lwc:if / lwc:elseif / lwc:else

<template lwc:if="{isLoading}">
    <lightning-spinner></lightning-spinner>
</template>
<template lwc:elseif="{error}">
    <c-error-panel error="{error}"></c-error-panel>
</template>
<template lwc:else>
    <c-data-view data="{records}"></c-data-view>
</template>

lwc:spread for dynamic attributes

<lightning-button lwc:spread="{buttonAttributes}" onclick="{handleClick}"></lightning-button>

get buttonAttributes() {
    return { label: this.label, variant: this.isImportant ? 'brand' : 'neutral', disabled: this.busy };
}

Event handlers stay outside the spread — onclick in the spread won't work.

lwc:on for dynamic event handlers (Spring '26) 

<button lwc:on="{eventHandlers}">Click</button>

eventHandlers = {
    click: this.handleClick.bind(this),
    focus: this.handleFocus.bind(this)
};

Event names lowercase, no on prefix. Always .bind(this) or arrow.

Complex template expressions (beta)

{!isLoading}, {count > 0}, {user?.profile?.name}, {value ?? 'default'}, {a && b}, ternary. Beta in 66.0 — prefer getters for anything non-trivial, but simple negation / optional-chaining inline is fine.

LWC project specifics

LWC wire, events, Flow, Jest, SLDS 2

Wire patterns

Always store the wire result for refreshApex:

wiredAccountsResult;

@wire(getAccounts, { searchTerm: '$searchTerm' })
wiredAccounts(result) {
    this.wiredAccountsResult = result;
    const { data, error } = result;
    if (data) { this.accounts = data; this.error = undefined; }
    else if (error) { this.error = this.reduceErrors(error); this.accounts = []; }
}

async handleRefresh() { await refreshApex(this.wiredAccountsResult); }

@wire for reactive reads. Imperative Apex for actions, DML, conditional/expensive loads.

Custom event patterns

Default to maximum encapsulation (bubbles: false, composed: false). Open it up only when crossing shadow boundaries. 

this.dispatchEvent(
    new CustomEvent('select', {
        detail: { recordId: this.recordId }
    })
);

Always copy non-primitives in detail ({ ...this.formData }) — never pass a live reference.

Event names: lowercase, no prefix, no underscores. recordchange, not onRecordChange or record-change.

LMS vs custom events: parent-child = custom events. Sibling components on different pages or LWC↔Aura = Lightning Message Service.

Flow hard-stops

Curated Salesforce reference: docs/flow-templates/ — upstream SKILL.md + Amplitude deltas (bypass flags, fault paths, bulk verification).

  • Never loop over $Record. The platform already handles bulk batching — use $Record directly. $Record__c doesn't exist.
  • No DML elements inside a Loop. Build a collection inside the loop, run a single Create/Update/Delete on the collection after.
  • No Get Records inside a Loop. Move it before the loop with an IN filter, then use a Collection Filter inside for in-memory lookups.
  • No hard-coded Salesforce IDs. Use Custom Metadata records storing DeveloperNames, resolve at runtime.
  • No unfiltered Get Records on large objects (Account, Contact, Opportunity, Task, Event). Filter on an indexed field.
  • Every DML element needs a fault path. Connect a fault connector that handles failure deliberately — log, notify, rollback.
  • Don't store all fields with storeOutputAutomatically=true on a Get Records against a wide object. Specify only fields the flow reads.
  • Each Screen element breaks the transaction boundary. DML committed before a screen cannot be rolled back after. For multi-step forms: collect all input across screens first, run DML after the final screen — or use Roll Back Records deliberately.

When NOT to build a flow

Before writing one, confirm a simpler tool can't solve it:

  • Formula field — derived value on a record, no automation needed.
  • Validation Rule — block a save with an error message.
  • Roll-Up Summary — sum/count/min/max on a master-detail parent.
  • Approval Process — multi-step approval routing.

Escalate to Apex (invocable) when the task involves: >5 nested decision branches or >20 elements; complex math/regex/string manipulation; HTTP callouts with retry; partial-commit transactions with savepoints; 10K+ records per run.

Flow type decision

Need Flow type
Update the same record before save Before-save record-triggered (fastest, no DML)
Update related records, send emails, callouts After-save record-triggered
User interaction (form/wizard) Screen flow
Triggered from Apex / Process Builder / button Autolaunched
Recurring on a schedule Scheduled flow (preferred over Schedulable Apex)
Triggered by a platform event Platform-event-triggered

Flow conventions and performance

Variable prefixes

Prefix Use
var_ regular variables (var_AccountName)
col_ collections (col_ContactIds)
rec_ record variables (rec_Account)
inp_ inputs (inp_RecordId)
out_ outputs (out_IsSuccess)

Element names

PascalCase_With_Underscores, descriptive. Check_Account_Type not Decision_1. Default names like Decision_1 are a DONE-blocker — rename before deploy. Subflows: Sub_ prefix (Sub_LogError, Sub_SendEmailAlert).

Descriptions matter

Every flow needs a description. Agentforce reads flow and element descriptions to decide what to invoke.

Flow-level template:

[PURPOSE]. [TRIGGER]. [OBJECTS INVOLVED]. [WHAT CHANGES]. [DEPENDENCIES if any].

For complex Decision / Loop / Get Records / Assignment elements:

WHY: [business reason]
WHAT: [technical summary]
EDGE CASE: [special handling]

Performance patterns

Collection Filter (avoids N+1 SOQL). Don't query inside a loop — query once with IN, filter in memory: 

1. Get Records: ALL Contacts WHERE AccountId IN {!col_AccountIds}
2. Loop Accounts:
   → Collection Filter: Contacts WHERE AccountId = {!current_Account.Id}
   → Process (zero SOQL per iteration)

Transform over Loop (when no per-record logic needed). 30-50% faster than equivalent Loop + Assignment. Loop only when each record needs different processing.

$Record direct access. In a record-triggered flow, never re-query the trigger object: 

✅ Decision where {!$Record.StageName} equals "Closed Won"
❌ Get Records → Account WHERE Id = {!$Record.Id}

Custom Metadata for thresholds & toggles. Prefer formula-reference path (zero SOQL): 

{!$CustomMetadata.Flow_Settings__mdt.Discount_Threshold.Numeric_Value__c}

Never store Salesforce IDs in CMDT — store DeveloperNames, resolve via Get Records at runtime.

Error handling — three tiers

  1. Input validation (pre-execution). Check null/empty required values via a Decision at the top.
  2. DML fault paths (during execution). Every Create/Update/Delete has a fault path capturing {!$Flow.FaultMessage}.
  3. Multi-step rollback (post-failure). If step 3 fails, delete records created in steps 1–2. Single transaction = automatic rollback, but explicit Delete steps make intent clear.

Data-load bypass

For data migration / bulk loads, use a Custom Permission entry condition rather than a Decision inside the flow:

  1. Create Bypass_<FlowName>__c Custom Permission.
  2. Assign to a "Data Load Bypass" Permission Set.
  3. Flow entry condition: NOT({!$Permission.Bypass_<FlowName>__c}).

Flow deploy + activation

New flows or new versions deploy as Draft initially. Activation is a separate, deliberate step the user must approve. Bulk-test 251+ records before requesting activation.

Flow anti-pattern checklist (run before DONE)

  • [ ] No hard-coded Salesforce IDs in the flow XML
  • [ ] No DML or Get Records inside a Loop
  • [ ] No looping over $Record
  • [ ] No unfiltered Get Records on large objects
  • [ ] Every DML element has a fault path connector
  • [ ] Variables follow var_ / col_ / rec_ / inp_ / out_ prefixes
  • [ ] Elements have descriptive PascalCase names (no Decision_1)
  • [ ] Flow-level description fits the PURPOSE/TRIGGER/OBJECTS/CHANGES template
  • [ ] Get Records specifies field list (no full-record auto on wide objects)

Metadata landmines

Curated Salesforce reference: docs/metadata-templates/ — 8 per-asset-type bundles (custom-object, custom-field, flexipage, permission-set, validation-rule, list-view, custom-tab, custom-application) with project deltas.

These are silent deploy failures. Catch them while writing the XML, not after the deploy report.

Permission sets

  • Required fields must NEVER appear in fieldPermissions. This silently fails the deploy. Check the field's <required> before adding to a perm set.
  • Tab visibility: three states only — Visible, Available, None.
  • Custom tabs need the __c suffix in the perm set entry; standard object tabs use the standard- prefix.
  • ViewAllData / ModifyAllData require security review — flag rather than auto-add.
  • Layered permission updates when a task adds new code:
    • classAccesses for new Apex classes
    • objectPermissions for new objects
    • fieldPermissions for new fields (subject to required-field rule above)

Custom objects

  • New custom object MUST have a layout. Without one it's unusable in the UI. Include the layout-meta.xml in the deploy.
  • Sharing model: ReadWrite by default; ControlledByParent if Master-Detail relationships exist.
  • Max 2 Master-Detail relationships per object (Salesforce platform limit).
  • No <fullName> in the root element of the object-meta.xml.
  • No reserved words as API names.

Custom fields

  • Universal requirements: every field needs <fullName>, <label>, <description>, and <inlineHelpText>.
  • Master-Detail fields: NEVER include <required>, <deleteConstraint>, or <lookupFilter>. They're always required and cascade by design.
  • Roll-Up Summary fields: NEVER add <precision> or <scale>. Both <summaryForeignKey> and <summarizedField> must use qualified format: ChildObject__c.FieldName__c.
  • Formula fields: <type> is the result type (e.g. Number), NEVER "Formula". All formula content in <![CDATA[ ... ]]> wrappers.
  • Picklists: default to <restricted>true</restricted> unless explicitly unrestricted.
  • Numeric: ensure scale ≤ precision ≤ 18.

Validation rules

  • errorMessage ≤ 255 characters.
  • fullName ≤ 40 characters, letters/numbers/underscores only, no leading underscore, no consecutive underscores.
  • Common formula gotchas:
    • TEXT() should not wrap text fields — remove it.
    • CASE() requires an even number of parameters with a default value last.
    • VALUE() is only for text fields; remove for numbers.
    • Date functions (DAY, MONTH) need Date fields, not Datetime.
    • ISPICKVAL() is required for picklist equality checks.
    • ISCHANGED() detects modifications.
  • Formulas containing XML tags must use CDATA sections.

FlexiPages

  • Must bootstrap via sf template generate flexipage — can't author from scratch reliably (strict deploy validation on structure).
  • Field references use Record.{FieldName} format.
  • Unique identifiers and region names required across the entire file (no duplicates).
  • Multiple components in the same facet combine into ONE region with multiple itemInstances.

Custom tabs

Three tab types, each with strict element allowlist (extra elements = deploy fail):

  • Object Tabs ({ObjectApiName}.tab-meta.xml): only <customObject>true</customObject> + <motif>. Don't add <label>, <sobjectName>, <name>.
  • Web Tabs ({TabName}.tab-meta.xml): <customObject>false</customObject>, <label>, <motif>, <url>, <urlEncodingKey>UTF-8</urlEncodingKey>. Optional: <description>, <frameHeight>.
  • Visualforce Tabs ({TabName}.tab-meta.xml): <customObject>false</customObject>, <label>, <motif>, <page>.

Root element must be <CustomTab> with xmlns="http://soap.sforce.com/2006/04/metadata".

Custom applications

navType decision: Standard (default) for general business apps. Console only when the workflow requires managing multiple related records simultaneously in split-view / multi-tab workspace.

List views

  • File at force-app/main/default/objects/<ObjectName>/listViews/<fullName>.listView-meta.xml.
  • Use correct API names for custom fields (Status__c), standard field names (NAME).
  • Filter operations must match field types.

FLS strategy

When adding a field, decide explicitly (BLOCKER if unspecified):

  • Permission-set vs. profile controlled. Match the org's existing convention for similar features.
  • For new fields read by Apex/LWC, add fieldPermissions to the relevant permission set under force-app/main/default/permissionsets/.
  • For fields written via DML, ensure they're editable (not just readable) for the target user.

Managed package field retrieval

If you need to reference a managed package field that isn't local:

sf project retrieve start --metadata "CustomField:<Object__c>.<Field__c>"

Do not use --output-dir. Do not query FieldDefinition via SOQL.

Deployment ordering

Curated Salesforce reference: docs/deploy-references/ — sf project deploy patterns, validation-only vs deploy-and-test, test-level selection, deploy-report parsing.

When a single change touches multiple metadata types, deploy in this order to avoid dependency and FLS failures:

Phase Metadata
1 Custom objects / custom fields
2 Permission sets / permission set groups
3 Apex classes / triggers
4 Flows — deploy as Draft
5 Flow activation / FlexiPage / layout / post-verify

Why each step:

  • Objects/fields first — Apex, Flows, permsets, and FlexiPages all reference these by API name. Deploy them later and you'll get INVALID_FIELD / INVALID_TYPE failures from the dependent metadata.
  • Permsets before Apex — if Apex references a new field, the running deploy user needs FLS to compile the class. (FLS isn't enforced at compile time in every case, but it is for WITH USER_MODE queries and @AuraEnabled cacheable methods touching the field.)
  • Apex before Flows — Flows that invoke Apex actions fail validation if the class isn't deployed yet.
  • Flows as Draft, then activate — deploying an active Flow that fails its own validation can take down a running process. Deploy inactive, run anonymous Apex / a manual test, then activate as a separate deploy. Copado does this automatically; manual sf project deploy start does not.
  • FlexiPage / layout last — these reference fields, related lists, and components; they need everything else in place.

For Copado user stories: the full-permission git setting is enabled, so Copado deploys perm sets as-is. Just promote the PermissionSet file alongside the new objects/fields it references — no need to manually expand the user story scope. (Historical: before that setting, Copado stripped perm-set XML by user-story scope; see [feedback_copado_permset_scope] for the obsolete failure mode.)

Existing project frameworks (cite, don't restate)

These are the project's bedrock conventions. Don't reinvent — read CLAUDE.md for the full context. Quick references:

  • Trigger framework: all triggers use TriggerDispatcher + *TriggerHandler (implements TriggerInterface) + *TriggerHelper. Recursion guards via static booleans (stopRecurssionBefore, etc.). Disable via TriggerSettings__c custom setting in tests.
  • Test data: AMPTestDataFactory with build* (no DML) / create* (inserts) convention. Disable triggers in @TestSetup via TriggerSettings__c.
  • Callouts: Named Credentials (Workato / WorkatoSandbox, Gainsight / GainsightSandbox, etc.). Sandbox routing via isSandbox() helper. ApiCalloutLogger.flush() after the last callout in a queueable.
  • Auth taxonomy: docs/salesforce-taxonomy.md is authoritative for field definitions, metric calculations, GTM segmentation. Confirm fields under force-app/main/default/objects/<Object>/fields/ before relying on names.

Reuse and pattern continuity

The fastest way to introduce bugs is to write a parallel implementation of something that already exists. Before adding new code, scan for existing helpers, services, or patterns and use them.

Reuse before reinventing

  • Helpers / utilities: Grep the codebase before writing a new helper. Examples: AMPTestDataFactory (test data), ApiCalloutLogger (callout audit), TriggerDispatcher (trigger orchestration), isSandbox() (env routing). If one exists, use it — don't write a parallel version.
  • Patterns: if the codebase has an established way to do something (trigger framework, Workato chunked queueable, named-credential isSandbox() switch), follow it. Don't introduce a new approach without an explicit reason.
  • Cite the existing example when reviewing: a "pattern drift" finding without a concrete file:line reference to the established approach is not actionable. Always quote the existing example.

Intra-change DRY

  • If the same logic appears three times in a single change (three similar branches, three near-duplicate methods, three repeated SOQL queries), collapse it into one helper. Two is a coincidence; three is duplication.
  • If a new helper is being added, ask whether it should live as a method on an existing utility class instead of standalone.

When a parallel approach IS justified

Sometimes divergence is intentional (different security model, different scale envelope, deprecation in progress). In those cases the change must include a one-line comment explaining why the existing approach was not reused, so future readers don't try to merge them.
