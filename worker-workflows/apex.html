Verify org target shell gate

  • sf config get target-org
  • must be spencerdev — anything else → BLOCKER

Prod is read-only (CLAUDE.md hard rule). Never deploy elsewhere without an explicit in-task instruction this turn.

Implement write

  • force-app/main/default/classes/<Name>.cls
  • force-app/main/default/classes/<Name>.cls-meta.xml
  • force-app/main/default/triggers/<Name>.trigger (if applicable)

No comments, no docstrings, no error handling beyond what the task specified. Use Edit on existing files — match the file's existing style, don't impose a template's shape on it.

Test + coverage gate shell gate

  • force-app/main/default/classes/<Name>Test.cls (+ meta)
  • sf apex run test --class-names <Name>Test --result-format human --wait 5
  • all tests pass · coverage ≥ 80% on changed class
  • playbook test rules applied (one behavior per test, Assert class, 251+ records)

A passing run that violates test-discipline rules is still a fail — iterate.

Code analyzer shell gate

  • sf code-analyzer run --target <files>
  • remediate sev1/sev2 before deploy

Skip with note if tooling unavailable. Do not deploy with known sev1/sev2 in scope.

Deploy targeted shell

  • sf project deploy start --source-dir <.cls> --source-dir <.cls-meta.xml> --source-dir <Test.cls> --source-dir <Test.cls-meta.xml>
  • sf project deploy report --job-id <id> (on failure)

Always include -meta.xml siblings and the test class. On failure: read the error, fix root cause, re-deploy. Never retry an identical deploy. Never --no-verify or skip hooks.

Commit per-task shell

  • git add <explicit paths only>
  • git commit -m "[T-NNN] <TODO.md description>"

Never git add -A or git add . — other workers may have uncommitted changes on this branch. Stage explicitly by path. Never push.

Verify before claiming DONE read

  • deploy report's Status: Succeeded line
  • test result's pass count + coverage line
  • git log -1 --oneline to confirm the commit landed

Every number in the DONE message quotes fresh output. No hedging language.

Return DONE / BLOCKER / FAILED gate

  • DONE: task ID · files touched · deploy ID + status · test pass count · coverage % · commit SHA · playbook sections cited by anchor
  • BLOCKER: <q> | CONTEXT: <why> | NEED: <unblocker> — revert partial edits, no commit
  • FAILED: <reason> after exhausting reasonable fixes — revert partial edits, no commit

Receiving code review findings (CHANGES_REQUESTED)

When the orchestrator re-delegates with review findings, follow this protocol — same gate, plus a finding-by-finding response in the DONE message:

Read every finding before changing code read

  • understand the full set first — don't fix in isolation
  • BLOCKER + MAJOR must be fixed or explicitly justified for dismissal
  • MINOR: fix if cheap, otherwise log "MINOR not addressed: <reason>"

Follow-up commit on top shell

  • git commit -m "[T-NNN] review fixes <round>"

Don't amend. Don't squash. Reviewer diffs the new commit cleanly.

Re-run the verification gate shell

Tests, coverage, deploy, analyzer — same evidence requirements as round 1.

Report DONE with finding-by-finding response gate

Format: [BLOCKER] file:line — Fixed: <what> · [MAJOR] ... — Fixed: ... · [MINOR] ... — Not addressed: <reason>. Then standard DONE evidence below.

Hard rules

Rule Why
Never deploy to non-spencerdev orgs CLAUDE.md "Production is read-only" hard rule.
Never push or merge Commits go on the feature branch only. User reviews the diff before merge.
Never git add -A / git add . Other workers may have uncommitted changes — stage explicitly by path.
Never modify .pm/ files Orchestrator's lane.
Never modify LWC / Aura / object / flow metadata BLOCKER → orchestrator routes to the right worker.
Never stall waiting for input Ambiguity → BLOCKER → return immediately.
Never claim DONE without fresh evidence Every number (deploy ID, test count, coverage %, commit SHA) from output read this iteration.
Never --no-verify or skip hooks Pre-commit hooks (Husky + lint-staged) run on staged files for a reason.

What this worker can / cannot do

Can

  • Write or edit Apex classes, triggers, queueables, batches, schedulables, REST resources, invocables
  • Create / extend test classes and run them with coverage
  • Run sf code-analyzer and remediate sev1/sev2
  • Deploy targeted to spencerdev and commit per-task on the feature branch
  • Read docs/data-references/, debug-references/, deploy-references/

Cannot

  • Edit LWC / Aura / flows / objects / fields / permission sets / layouts
  • Deploy to any org other than spencerdev
  • Push to a remote, merge, or auto-create PRs
  • Modify .pm/ orchestrator state files
  • Skip pre-commit hooks or use --no-verify
  • Stall asking for input — ambiguity returns BLOCKER
