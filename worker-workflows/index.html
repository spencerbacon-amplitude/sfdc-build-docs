SFDC worker workflows 7 pages, one per worker

Sourced from ~/.claude/agents/sfdc-*-worker.md · paired with the combined view at docs/worker-workflows.html · matches post sf-skills-mining-pass-2 (master 5c16b38d).

Each card opens a dedicated page documenting one worker end-to-end — numbered step cards, file paths, tag pills, hard rules. For a side-by-side comparison of Apex + LWC + Review (the highest-volume builders), see the combined view card at the bottom.

🟧 Apex worker
Writes Apex classes, triggers, queueables, batches, and test classes. Owns the 80%-coverage gate and the code-analyzer pass.
~/.claude/agents/sfdc-apex-worker.md
🟪 LWC worker
Writes LWC and Aura bundles. Owns the lint + jest + targeted-deploy gate. Reads the PICKLES design doc when one exists.
~/.claude/agents/sfdc-lwc-worker.md
🟩 Flow worker
Authors record-triggered, screen, autolaunched, scheduled, and platform-event flows. Always deploys as Draft — activation needs explicit user approval.
~/.claude/agents/sfdc-flow-worker.md
🟫 Metadata worker
Schema and metadata: custom objects, fields, validation rules, layouts, permission sets, record types, flexipages, tabs, applications, list views. Watches the metadata landmines.
~/.claude/agents/sfdc-metadata-worker.md
🟦 Research worker
Read-only. Fans out searches in parallel, returns full file contents so downstream workers don't re-read. Invoked at kickoff and on demand.
~/.claude/agents/sfdc-research-worker.md
🟦 Review worker
Read-only. Runs after every builder DONE in one of two modes — spec (does it meet the ACs?) or quality (does it follow the playbook?). Both must approve.
~/.claude/agents/sfdc-review-worker.md

How these pages relate to the prompts

Each per-worker page is a synthesis of its source prompt at ~/.claude/agents/sfdc-<x>-worker.md, re-shaped into a numbered visual flow. The prompts stay authoritative — if the page and the prompt disagree, the prompt wins. The link audit at .pm/worker-docs-pages/LINK-AUDIT.md (gitignored) keeps the step-by-step mapping fresh.
