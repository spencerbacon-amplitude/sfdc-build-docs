Two architectural rules drive everything:
  1. Permission-decision hooks must invoke a .ps1 file via powershell -NoProfile -File. Inline PowerShell commands that emit permissionDecision JSON are silently ignored on Windows. Side-effect-only hooks (Slack pings, log writes) can stay inline.
  2. Hook payloads carry no skill context. Hooks key off file paths, command patterns, and .pm/<slug>/ state — never on "which skill is running." This is why quick-build inherits all of drive-build's hook coverage automatically: same file paths, same state shape.

SessionStart hooks fires once at boot

S1 · Worktree sweep

script · ~/.claude/scripts/worktree-sweep.ps1

What it does: Scans .claude/worktrees/ for "claude agent (pid N)" locks where PID N is dead. If the worktree is clean AND the branch is merged to master, removes the lock + worktree + branch via safe git branch -d. Prints breadcrumbs (parent chat UUID + agent description) so you can resume context if needed.

Cost / risk: ~1–2s at boot · Low risk. Never -D force-delete; never touches paths outside .claude/worktrees/.

S2 · .pm/<slug>/ + spec resume scanner

inline PowerShell · prints "where was I" context block

What it does: Walks .pm/*/ dirs that contain a TODO.md. For each one, prints spec frontmatter (first 15 lines), TODO.md (first 40 lines), LOG.md (last 15 lines), and the count of "awaiting" strings in QUESTIONS.md. Also lists orphan in-progress specs that have no .pm/<slug>/ dir.

Cost / risk: ~0.5s. Output is the resume context you see on every session boot. Read-only.

Heading text: renamed from ## Active drive-build: → ## Active build: on 2026-05-22 to be accurate for both quick-build and drive-build slugs.

S3 · Weekly doc-writer queue check

script · ~/.claude/scripts/weekly-doc-check.ps1

What it does: Once per ISO week (state file at ~/.claude/state/last-doc-check.txt), prompts review of the doc-writer queue for $AMP_JIRA_PROJECT (default GTMSYS).

Cost / risk: ~0.5s, fires once per week. Low risk · state-throttled.

Stop hooks fires when Claude turn ends

T1 · "Claude stopped" Slack notify

script · ~/.claude/scripts/notify-stop.ps1 · async

Silence rules (in order):

  1. Any .pm/*/QUESTIONS.md contains an "awaiting" string → silence (T2 will handle the ping).
  2. Any file under .pm/*/ was modified in the last 5 min → silence (active loop iteration).
  3. Otherwise → Slack-ping $AMP_SLACK_WEBHOOK with a random "bro" nickname.

What this catches: Drive-build dying silently with no Q written (no recent activity → ping). Stale in-progress specs (no Q, no recent activity → ping you to clean it up). Normal conversational stop with no active builds → ping.

What this silences: Active loop iterations between waves (recent .pm/ activity). Builds with an awaiting Q (T2 already pinged).

Replaced: Old "if any spec status is in_progress → silence" logic which had a blind spot for silently-dead builds.

T2 · QUESTIONS.md awaiting-count Slack notify

inline PowerShell · async

What it does: Scans .pm/*/QUESTIONS.md for "awaiting" string counts. Compares to last-pinged count file at .pm/<slug>/.last_pinged_count. If current count > last, Slack-pings: Hey <bro> (<branch>), N new Q(s) on builds: <slug-list>. Updates the count file.

Cost / risk: ~50ms async. Per-build count file prevents repeat-pings on the same questions.

Windows gotchas

Inline PowerShell silently fails for permission decisions. If you write a PreToolUse hook that emits { hookSpecificOutput: { permissionDecision: ... } } JSON inline, Claude Code on Windows ignores the output. You MUST invoke a .ps1 file via powershell -NoProfile -File <path>. Side-effect hooks (Slack pings, append-to-log) work inline fine.
The # character in inline PowerShell hook strings consumes everything after it. In a single-line command string inside settings.json, # starts a comment and silently no-ops the rest of the script. Never include # in inline hook bodies; use externalized .ps1 files for any script that needs comments.
Plan mode bypasses all PreToolUse permission hooks. If you're in plan mode and get prompted on every edit despite R1's allowlist, check plan mode first — it has a Yes/No-only prompt (no "Always allow") that's separate from the standard permission dialog.
